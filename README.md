# memos

## usememos/memos `v0.1.0` 迁移到 `gin-ninja` 改造难度评估

结论：**中高难度（约 3~8 人天，受接口规模、测试覆盖率与历史技术债影响）**。

### 主要工作量来源

1. **路由层改造（中）**
   - 将原 `gin` 路由注册方式迁移到 `gin-ninja` 的分组+typed handler 风格。
   - 示例：原 `gin` 常见写法是 `r.GET("/api/memo", handler)`（`handler` 通常为 `func(*gin.Context)`，`r` 可为 `*gin.Engine` 或 `*gin.RouterGroup`）；迁移后可写为 `ninja.Get(router, "/api/memo", listMemos)`，其中 `ninja` 及相关类型来自项目实际选用的 `gin-ninja` 包导入，`router` 为 `gin-ninja` 的 `*ninja.Router`，`listMemos` 通常为 `func(ctx *ninja.Context, in *ListMemosRequest) (*ListMemosResponse, error)`。
   - 需要逐个 API 校对 URL、Method、参数绑定和中间件挂载顺序。

2. **参数绑定与校验（中）**
   - 统一迁移 `ShouldBindJSON/Query/Uri` 等输入绑定写法。
   - 校验错误结构可能变化（如字段名、错误数组结构、HTTP 状态码），建议通过统一错误转换层维持旧响应格式（例如继续返回既有 `code/message` 结构）。

3. **中间件与上下文（中高）**
   - 认证、鉴权、CORS、日志、错误恢复等中间件链需要逐条验证。
   - `gin.Context` 读写习惯若变化，会影响 handler 逻辑和错误处理分支。

4. **文档与测试回归（中高）**
   - 若项目依赖 OpenAPI/Swagger，需要同步验证生成结果。
   - 建议针对登录、笔记 CRUD、标签/资源上传等核心路径做回归测试。

### 推荐最小风险迁移策略

1. 先保留原目录结构，仅替换路由注册入口（不改业务层）。
2. 分模块迁移（auth -> user -> memo -> resource），每模块迁移后立刻回归。
3. 保持响应体结构不变，先兼容再优化。
4. 最后统一整理中间件和错误码映射。

### 实施边界

- **目录结构保持不动**：不重组 `service / store / model` 等现有业务代码组织。
- **第一阶段只动 HTTP 入口层**：仅调整路由注册、handler 签名、参数绑定和统一错误出口。
- **业务逻辑原样复用**：不在首阶段改 DAO、核心 service、领域对象与数据库访问逻辑。
- **迁移前先完成接口清单盘点**：把现有 API 按 `auth`、`user`、`memo`、`resource` 四个模块归类，记录 URL、Method、入参来源、鉴权要求、响应结构。

### 第一阶段目标

在不改变业务实现的前提下，引入一层新的 `gin-ninja` 路由适配入口，做到：

- URL 不变
- HTTP Method 不变
- 状态码不变
- 响应 JSON 字段不变
- 中间件顺序与语义尽量不变

### 路由适配层实施方式

建议新增一层单独的路由注册入口，用于承接 `gin-ninja` 的路由定义，但内部仍调用原有业务逻辑：

1. 为各模块建立新的注册函数，集中维护 `gin-ninja` 路由映射关系。
2. handler 先做“薄适配”，只负责：
   - 解析 path/query/body 参数；
   - 调用旧 service 或旧 handler 核心逻辑；
   - 把错误与响应包装回旧协议格式。
3. 公共错误输出单独收口，避免每个接口各自处理兼容逻辑。
4. 对旧代码中依赖 `gin.Context` 的用户态、trace、事务、request id 等上下文数据，优先通过适配层桥接，而不是立即重写业务代码。

### 分模块迁移清单

#### 1. auth 模块

- 迁移登录、登出、会话校验、身份鉴权相关路由。
- 对齐认证中间件的执行顺序与上下文注入方式。
- 回归重点：
  - 登录成功/失败响应体结构是否一致；
  - 未登录、登录过期、权限不足时的状态码和错误体是否一致；
  - cookie / token / session 读写行为是否一致。

#### 2. user 模块

- 迁移用户查询、用户更新、用户状态变更等路由。
- 保持用户 ID、分页参数、筛选参数的绑定方式兼容。
- 回归重点：
  - 当前用户信息读取是否正常；
  - 管理员与普通用户权限边界是否一致；
  - 更新类接口的校验错误格式是否一致。

#### 3. memo 模块

- 迁移 memo 的查询、创建、更新、删除及相关筛选路由。
- 重点处理 query 参数、分页、排序、过滤条件的绑定兼容。
- 回归重点：
  - CRUD 主链路是否正常；
  - 列表过滤、分页、权限过滤是否正常；
  - 业务异常、资源不存在、越权访问时的返回是否一致。

#### 4. resource 模块

- 迁移资源上传、下载、列表、删除等路由。
- 保持 multipart/form-data、文件流输出和 content-type 行为不变。
- 回归重点：
  - 上传大小限制、文件类型校验是否一致；
  - 下载响应头、文件名、媒体类型是否一致；
  - 资源访问权限与错误处理是否一致。

### 每个模块的执行节奏

每个模块都按以下顺序推进，避免大批量同时改动：

1. 梳理该模块旧接口清单。
2. 补齐该模块的 `gin-ninja` 路由注册。
3. 用适配层接入旧业务逻辑。
4. 完成模块级回归验证。
5. 确认稳定后冻结该模块，再进入下一个模块。

### 协议兼容要求

迁移期间，以下内容建议视为硬约束：

- 请求路径保持不变；
- 请求方法保持不变；
- 状态码保持不变；
- 响应 JSON 字段名和层级保持不变；
- 旧前端或外部调用方无需联动修改；
- 输入绑定差异由适配层消化；
- 校验失败、业务异常、鉴权失败统一走兼容错误映射。

### 高风险点优先验证

#### 1. 中间件顺序

- CORS、日志、恢复、鉴权、权限校验的挂载顺序必须与旧实现逐项比对。
- 若中间件执行时机变化，可能导致响应头、上下文数据或错误处理分支不一致。

#### 2. 上下文桥接

- `gin.Context` 迁移到 `ninja.Context` 后，要重点核对：
  - 用户身份信息注入；
  - request-scoped 数据透传；
  - trace / request id；
  - 事务对象或数据库上下文绑定。

#### 3. 输入绑定与错误结构

- `ShouldBindJSON / Query / Uri` 的行为差异要单独验证。
- 字段缺失、类型错误、非法枚举值等校验失败场景，必须继续返回旧格式。

#### 4. 文件与流式响应

- 文件上传下载、附件预览、二进制响应最容易出现兼容偏差，应优先验证。

### 每模块完成后的回归标准

- 核心接口可正常访问；
- 响应体结构无破坏性变化；
- 认证、鉴权、跨域、日志、异常恢复链路正常；
- 关键异常分支与旧行为一致；
- 若启用 OpenAPI / Swagger，则生成结果可用且不影响既有调用。

### 最后统一收口

全部模块迁移完成后，再进行第二阶段整理：

1. 合并重复的路由注册模式；
2. 收敛公共输入输出模型；
3. 统一错误码与错误结构映射；
4. 梳理公共中间件配置；
5. 在兼容稳定后，再考虑进一步利用 `gin-ninja` 的 typed handler、文档生成与校验能力做优化。

### 建议交付物

为了让迁移过程更可控，建议同步维护以下产物：

- 一份按模块拆分的 API 清单；
- 一份模块迁移状态表；
- 一份回归检查表；
- 一份兼容性差异记录表。
